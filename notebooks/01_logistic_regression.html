Logistic Regression

Business question: Who will churn / convert / default? Output: Probability 0–1 Dataset: customers.csv (800 rows, 12 features) Use case: Churn propensity model

This guide walks through the complete analysis pipeline for a logistic regression churn model — from raw data to a calibrated, deployment-ready scorer. Each stage shows the workflow steps an analyst takes, followed by the exact outputs you should expect and how to read them.

Stage 01

Data Loading & Exploratory Analysis

Before any modelling, the analyst loads the dataset and answers three questions: What does the target variable look like? Are features usable? Are there data quality issues that will corrupt the model?

  • Load and inspect the dataset Read the CSV into a dataframe. Check shape, column names, and data types. Confirm the target column (churned: 1/0).What to watch: columns that are IDs, dates, or free text — these need special handling or exclusion.
  • Check the target variable distribution Count the split between churned (1) and retained (0). Calculate the base event rate as a percentage.This tells you immediately whether class imbalance is a problem.
  • Profile each feature For numeric features: min, max, mean, median, % missing. For categoricals: unique values, most frequent.Flag features with >30% missing, extreme outliers, or near-zero variance.
  • Visualise feature distributions by churn status Plot histograms split by churned vs retained. Look for features where the two distributions clearly separate — these will be strong predictors.Strong visual separation = high feature importance in the model.
  • Check correlations between features Produce a correlation matrix for numeric features. Flag pairs with |r| > 0.7 as multicollinear — these need regularisation (Ridge) or one to be dropped.Multicollinear features don't cause errors but make coefficient interpretation unreliable.

Stage 02

Feature Engineering & Preprocessing

Raw data is never model-ready. The preprocessing pipeline transforms features into the format the model expects: numeric features standardised, categoricals encoded, missings imputed. Standardisation is mandatory for logistic regression — without it, coefficients are not comparable.

  • Separate target and features Remove the target column (churned) and any identifiers (customer_id) from the feature set. Identify numeric vs categorical columns programmatically.Always exclude IDs, timestamps, and post-event variables from features.
  • Impute missing values Numeric: replace with median (robust to outliers). Categorical: replace with most frequent value.Use median over mean for business data — spend and frequency metrics are usually right-skewed.
  • Standardise numeric features Subtract mean, divide by standard deviation (StandardScaler). All numeric features now have mean=0, std=1.This step is mandatory — without it, a coefficient of 0.02 on revenue (range 0–100K) is not comparable to a coefficient of 2.0 on age (range 18–80).
  • Encode categorical features One-hot encode categorical columns (channel, region, tier, plan_type). Each category becomes a binary 0/1 column.Drop the first category from each group to avoid the dummy variable trap (perfect multicollinearity).
  • Bundle into a Pipeline Wrap all steps into a single sklearn Pipeline object. This ensures the same transformations are applied identically to train, validation, and test data — preventing data leakage from the preprocessing steps.

The standardisation rule After scaling, every numeric feature has the same unit. A coefficient of +0.45 on support_calls means the same magnitude of effect as a coefficient of -0.45 on tenure_months. Without scaling, a coefficient on revenue (raw units) would look tiny just because the scale is large.
Stage 03

Train / Validation / Test Split

The data must be divided into three non-overlapping sets before any modelling. This is a governance step, not just a technical one — it is the only way to get an honest estimate of production performance.

  • Hold out a test set immediately Set aside 20% of data as the test set. This set is locked — it will not be seen until the final evaluation. Stratify by the target so both sets have the same churn rate.If the analyst reports performance "on the training data", that is not a valid performance estimate.
  • Create a validation set from training data Split the remaining 80% into 80% train / 20% validation. The validation set is used for hyperparameter tuning (choosing regularisation strength C).Alternatively: use cross-validation (5-fold) instead of a fixed validation set.
  • Confirm event rates are preserved Check that churn rate is approximately equal across train, validation, and test. Large differences indicate a stratification problem.Typical variance: ±2–3 percentage points is acceptable.

Common mistake — peeking at the test set Any decision made after looking at test set performance (including threshold choice, feature selection, or hyperparameter tuning) invalidates the test set. The test set should be opened exactly once — at the end. This is the single most important governance control in model validation.
Stage 04

Train Model & Interpret Coefficients

This is the core analytical step. The model is trained on the training set, and the resulting coefficients are read as business drivers. Standardised coefficients are the primary output for stakeholder communication — they show which features matter and in which direction.

  • Select regularisation type and strength Default: Ridge (L2 penalty, C=0.1). The analyst may tune C on the validation set — smaller C = stronger regularisation = simpler model. L1 (Lasso) is used if feature selection is the goal.For a first model, C=0.1 with L2 is a safe, stable starting point.
  • Fit the pipeline on training data The pipeline runs preprocessing then trains the logistic regression. The model learns the weight (coefficient) for each of the 22 features that minimises prediction error on the training set.
  • Extract and interpret coefficients Retrieve the coefficient vector from the trained model. Because features are standardised, coefficients are directly comparable: larger absolute value = stronger driver. Positive = increases P(churn), negative = decreases P(churn).Convert to odds ratios: exp(coefficient). An odds ratio of 1.5 means the odds of churn increase by 50% for each 1 standard deviation increase in that feature.
  • Build the coefficient chart Sort features by coefficient value. Plot a horizontal bar chart — red for positive (churn drivers), blue for negative (churn protectors). Annotate each bar with the odds ratio.This chart is the primary deliverable for a CMO or retention team briefing.
  • Sanity-check the direction of each coefficient Does each coefficient make business sense? Higher support calls should increase churn risk. Higher tenure should reduce it. Any counter-intuitive result needs investigation before presenting results.

Reading the coefficients — what to tell stakeholders
  • Tenure is the most powerful protector. Odds ratio 0.47 means a customer who is 1 standard deviation (18 months) longer-tenured has 53% lower odds of churning. Early engagement is the most valuable retention lever.
  • Support calls are the leading risk indicator. Each additional standard deviation of support calls (1.8 calls) nearly doubles churn odds. Customers with 4+ calls should be flagged for proactive intervention.
  • Logins are a proxy for engagement. Regular logins protect against churn — this is likely a leading indicator, not a cause. Correlate with product NPS to understand the mechanism.
  • Channel and tier effects are meaningful. Gold tier and Phone channel reduce churn risk — this may reflect both self-selection (happier customers upgrade) and genuine protection (higher service level).
  • ⚠ customer_id appearing as a top feature is a red flag — see Stage 8.
Stage 05

Model Performance & Calibration

Performance tells you how well the model ranks customers. Calibration tells you whether the probability outputs are trustworthy. Both must be checked before using the model for decisions.

  • Generate predictions on the test set Run the trained pipeline on the held-out test set. Produce probability scores (predict_proba) and binary predictions (predict, using 0.5 threshold).
  • Compute performance metrics AUC-ROC (ranking quality), Average Precision (summary of PR curve), Brier Score (probability accuracy). Also compute the confusion matrix at default threshold.AUC is the primary metric for ranking. Average precision is more informative with class imbalance.
  • Plot the calibration curve (reliability diagram) Bin predictions into 10 buckets (0–10%, 10–20%, etc.). For each bucket, plot the predicted probability vs actual churn rate. A well-calibrated model follows the diagonal.If P(churn)=0.7 but the actual churn rate for those customers is 40%, the model is overconfident — probabilities cannot be used directly for expected value calculations.
  • Plot score distribution by true label Histogram of predicted P(churn) separately for churners and retained customers. Good discrimination = two distinct distributions. Heavy overlap = weak model.

What these numbers mean for business use
  • AUC = 0.832 means the model correctly ranks a random churner above a random non-churner 83.2% of the time. A random model scores 0.5. Practical threshold: >0.75 is useful, >0.85 is strong.
  • Calibration is good — the model closely follows the diagonal. P(churn) = 0.6 genuinely means approximately 60% of those customers will churn. This makes the scores usable for expected value calculations (e.g., "expected revenue saved by contacting the top 200").
  • Recall at 0.5 threshold is 56% — the model misses 44% of churners at the default threshold. This is normal and expected — the right threshold should be chosen based on intervention capacity (Stage 6).
Stage 06

Threshold Selection — The Business Decision

The threshold is not a modelling decision — it is a business decision. It determines what fraction of customers get flagged for intervention, and what the tradeoff is between false alarms (wasted campaign cost) and missed churners (lost revenue). The optimal threshold depends entirely on the relative cost of each error type.

  • Plot Precision vs Recall vs Threshold Sweep the threshold from 0 to 1. At each value, compute precision (of those flagged, how many actually churn) and recall (of all churners, how many are caught). This shows the full tradeoff curve.Lower threshold = more customers flagged = higher recall, lower precision. Higher threshold = fewer flagged = higher precision, lower recall.
  • Calculate intervention capacity Ask the business team: "How many customers can we actually contact per week?" This capacity constraint defines the operating point — the threshold at which flagged volume matches capacity.Example: if retention team can contact 60 customers/week from 800 scored = flag top 7.5% = threshold ~0.55.
  • Compute expected ROI at the chosen threshold At threshold T: expected customers flagged × precision × expected churn prevention rate × average revenue per retained customer = expected revenue saved. Compare to cost of intervention (staff time, offer cost).
  • Document the chosen threshold and its rationale Record: threshold value, expected precision, recall, weekly flag volume, and the business assumption justifying the choice. This is a governance artefact.The threshold should be revisited quarterly — as the model is retrained, the score distribution may shift.

Stage 07

Cross-Validation & Stability

A single train/test split can be lucky or unlucky. Cross-validation repeats the evaluation across multiple splits to get a stable estimate of performance and its variance. Unstable performance across folds signals a fragile model.

  • Run 5-fold stratified cross-validation Split all 800 rows into 5 equal folds. Train on 4, test on 1 — repeat 5 times. Each fold produces one AUC score. Report mean ± standard deviation.Standard deviation >0.03 suggests the model is sensitive to which data it trains on — consider more regularisation.
  • Check coefficient stability across folds For each fold, extract the top 5 coefficients and their direction (sign). If the sign flips (e.g., tenure_months is sometimes positive and sometimes negative), the model is unstable — likely due to correlated features.Stable coefficients are required for the model to be governable — you need to commit to a story about what drives churn.
  • Compare C values on validation set Try C = 0.01, 0.1, 1.0, 10 on the validation set. Plot AUC vs C. Choose the value with the best validation AUC, or the most regularised model within 0.01 AUC of the best (principle of parsimony).

What stability means for governance
  • AUC std = 0.011 is excellent — the model will perform consistently even as the training window shifts over time.
  • All top-5 coefficients are stable in sign across all folds — this means the business story is reliable. "Tenure protects, support calls signal risk" holds regardless of which customers happened to be in the training set.
  • C=0.10 was chosen over C=1.0 despite marginal AUC — the principle is to use the simplest model that achieves acceptable performance. Simpler models are easier to retrain, monitor, and explain.
Stage 08

Leakage & Feature Audit

Feature leakage is the most common cause of models that perform brilliantly in the notebook and fail in production. It occurs when a feature contains information that would not be available at the time of prediction — meaning the model is "cheating" during training.

  • List all features and their data generation time For each feature, ask: "At the moment I need to score a customer, does this feature already exist in the database?" If the answer is "only after the outcome is known", it is a leaky feature.Example: "cancellation_request_submitted" is generated after the decision to churn — not before. Including it = 100% AUC, zero production value.
  • Check SHAP or coefficient values for suspicious features Any ID column, date column, or post-event flag appearing in the top features is a guaranteed red flag. customer_id should have zero predictive power — if it does, it is encoding group-level leakage.The customer_id effect in Stage 4 suggests some customer IDs are associated with a specific cohort that has a different churn rate — this is a data construction issue, not a genuine feature.
  • Rebuild without flagged features and compare AUC Remove all suspected leaky features. Retrain. If AUC drops dramatically (>5 points), the model was heavily reliant on leakage. If AUC holds, the leakage was minor and the model is genuinely predictive.A 10-point AUC drop after removing customer_id means the model was using ID-level cohort information. It needs to be redesigned.
  • Document the clean feature set Produce a final feature register: feature name, business meaning, source system, data freshness (how often is it updated), and confirmed availability at scoring time.

Stage 09

QA — Good vs Red-Flag Output

Use this as a sign-off checklist when reviewing model output produced by an analyst or data science team.

Good output — ready to proceed
  • AUC meaningfully above seasonal naive or rule-based baseline
  • Calibration plot follows the diagonal (±5 percentage points per bin)
  • Top features are available at scoring time and make business sense
  • Coefficient signs are directionally correct and stable across CV folds
  • Threshold chosen based on explicit FP/FN cost discussion, not defaulted to 0.5
  • customer_id, dates, and identifiers excluded from features
  • Test set was held out and used exactly once
  • Missing values handled with a principled strategy (not just dropped)
Red flags — investigate before sign-off
  • AUC = 1.0 or near-perfect — almost certainly leakage
  • AUC drops >8 points when removing any single feature — leakage dependency
  • customer_id or date columns in top features by coefficient/importance
  • Coefficient sign flips across CV folds for key features
  • Calibration curve far from diagonal — probabilities cannot be trusted
  • Performance reported on training data rather than held-out set
  • No mention of threshold — implies defaulting to 0.5 without justification
  • Feature engineering done after train/test split (preprocessing leakage)
Stage 10

Extension Directions

Once the baseline logistic regression is validated, these are the natural next steps — in order of priority for a strategy/PM context.

  • Segment-level performance analysis Break AUC and precision/recall down by channel, region, and tier. A model that performs well overall but poorly on Gold-tier customers (highest value) may need a separate model or feature set for that segment.
  • Compare against GBM baseline Train a LightGBM model on the same features and compare AUC. If the gap is <3 points, stick with logistic regression (interpretability advantage). If >5 points, the relationship is genuinely nonlinear and GBM is warranted.
  • SHAP values for individual explanations Use SHAP even on logistic regression to produce per-customer "why is this person high risk?" explanations for retention managers. Shows: "tenure -2.3, support_calls +1.8, logins -0.9".
  • Calibrated probability uplift sizing Multiply P(churn) by expected revenue × retention intervention success rate. Produces a "revenue at risk" per customer — enables prioritisation by expected value, not just score rank.
  • Monitoring dashboard design Track weekly: (a) score distribution shift (PSI), (b) precision at chosen threshold on resolved cases, (c) flag volume vs capacity. Trigger retraining when PSI >0.2 or precision drops >5 points.
  • Fairness audit Check AUC and precision separately by demographic groups (if available). EU AI Act and internal policies may require fairness metrics to be documented before deployment to customer-facing decisions.
