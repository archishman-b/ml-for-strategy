Pipeline walkthrough — analyst workflow steps, pre-populated outputs, and business interpretation at each stage.

Stage 01

Anomaly Detection Problem Setup

Anomaly detection finds observations that do not conform to expected patterns — without requiring labelled examples of what 'anomalous' looks like. This distinguishes it from classification: you cannot train a supervised model if you do not have labelled anomalies.

  • Choose the right method for the use case Isolation Forest: fast, scalable, works on high-dimensional tabular data — best general-purpose choice. One-Class SVM: better on small datasets with clear normal distribution. Autoencoder: best when normal behaviour has complex structure that can be learned. Statistical Z-score: for univariate outliers, no ML needed.Start with Isolation Forest — it is the most practically reliable method for business tabular data.
  • Decide on contamination rate The contamination parameter tells the model what fraction of data is anomalous (default 0.05 = 5%). This directly controls how many anomalies are flagged. Ask the business: 'What fraction of transactions do we expect to be fraudulent / anomalous?' This is a business input, not a data-driven decision.
  • Understand the output: anomaly score Isolation Forest returns an anomaly score: more negative = more anomalous. The threshold maps scores to binary labels. Scores are relative — they only mean something in comparison to other scores in the same dataset.

Stage 02

Train Isolation Forest & Score

Isolation Forest builds random trees that isolate points. Anomalies are isolated in fewer splits (they are unusual) — resulting in shorter average path length and a more negative anomaly score.

  • Fit IsolationForest(contamination=0.05, n_estimators=200) 200 trees provides stable scores. contamination=0.05 sets the threshold at the 5th percentile of anomaly scores — 5% of observations will be flagged.
  • Compute anomaly scores for all transactions Each transaction gets a score between -1 (most anomalous) and 0 (normal). Scores below the contamination threshold = flagged as anomaly.
  • Validate against known labels (if available) If you have any labelled anomalies, use them only for validation — never training. Compute precision, recall, and F1 on the labelled subset.

What this tells you
  • Precision of 44% means 44% of flags are genuine anomalies — the remainder are false alarms requiring analyst review. This is typical for anomaly detection: precision is usually low.
  • Recall of 73% means the model catches 73% of all labelled anomalies. The remaining 27% were not flagged — likely because their anomalous features are subtle or novel.
  • The top 3 flags show multiple simultaneous anomalous signals (late night, high velocity, device change) — these are the highest-confidence cases and should be prioritised for investigation.
  • Setting contamination=0.03 (matching known rate) would reduce false alarms but also reduce recall — a precision/recall tradeoff that the fraud/risk team must decide.
Anomaly detection precision is always low — this is expectedUnlike classification, anomaly detection operates without training labels. Precision of 30-60% is typical and acceptable if the cost of review is low relative to the cost of missed anomalies. Frame this to stakeholders as: 'The model surfaces the top 50 transactions most worth reviewing — we estimate 22 of those are genuine. Without the model, you would need to review all 1,000 to find those 22.'
Stage 03

QA & Monitoring

Anomaly detection models require active monitoring — the definition of 'normal' evolves as business patterns change.

  • Set contamination rate based on business input Do not default to 0.05 without asking the business. What fraction of transactions is genuinely expected to be anomalous?
  • Track precision and recall over time on resolved cases As analysts review flagged cases, record outcomes (true/false positive). Use this to track model drift and calibrate the contamination threshold.
  • Re-train when normal behaviour shifts A seasonal business (e.g., retail) will have very different 'normal' transaction patterns in November vs February. Seasonal models or rolling retrain windows are needed.
Output: QA Checklist Checklist
Good
  • Contamination rate set by business input (not default)
  • Model validated against any available labelled anomalies
  • Top flags profiled — multiple simultaneous anomalous signals = higher confidence
  • Monitoring plan: precision/recall tracked on resolved cases
  • Re-training cadence defined
Red flags
  • Default contamination=0.05 used without business validation
  • No labelled examples used for validation at all
  • Model treats every flagged case as confirmed anomaly (no analyst review)
  • Static threshold — not adjusted as normal behaviour evolves
  • Only anomaly score reported, not interpretable feature profile per flag
