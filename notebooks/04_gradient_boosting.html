Notebook 1.4

Gradient Boosting (XGBoost / LightGBM)

Business question: Best-performance classification on tabular data Output: Probability 0–1 Dataset: customers.csv (800 rows, 12 features) Use case: Production churn model · Real-time fraud scoring

This guide walks through the complete analysis pipeline — each stage covers workflow steps, followed by pre-populated example outputs and interpretation for a strategy or PM context.

Stage 01

Data & Model Selection Context

GBM should be selected after establishing an LR baseline and confirming the performance lift justifies the additional complexity and governance effort.

  • Establish LR baseline AUC Same data, same split as Notebook 01. Confirm LR AUC before proceeding.
  • Understand the boosting mechanism Sequential trees: each tree corrects the residuals of the current ensemble. Gradient descent in function space. The learning_rate controls how much each tree contributes.
  • Choose library: XGBoost vs LightGBM LightGBM: faster on large datasets (histogram-based leaf-wise splitting). XGBoost: more granular regularisation control. For most strategy work ≥10K rows: LightGBM. For smaller data or maximum AUC: XGBoost.
Output: Model Selection Rationale Console
LR baseline AUC: 0.832 GBM target: should exceed 0.850 to justify complexity Decision criteria: - AUC lift > 0.02: GBM justified - Interpretability required (credit/HR context): use LR + SHAP - Dataset < 5K rows: LR or RF often comparable to GBM - Production latency < 50ms: confirm serving infrastructure can handle GBM For this dataset: LR=0.832, GBM expected ~0.870 → proceed with GBM
Stage 02

Train with Early Stopping

Early stopping is the single most important technique for GBM — it prevents overfitting by stopping training when validation performance stops improving.

  • Split into train/validation/test 80/10/10 or 70/15/15 for GBM — needs a separate validation set for early stopping callbacks.The validation set is used by the early stopping callback — not for reporting final performance (that uses the test set).
  • Set learning_rate=0.05 and large n_estimators=1000 With early stopping, you set a large upper bound. The model will stop automatically at the optimal point.
  • Configure early_stopping_rounds=50 If validation AUC doesn't improve for 50 consecutive trees, stop training. This prevents both overfitting and unnecessary computation.

What this tells you
  • Train AUC (0.968) vs Val AUC (0.874) — the gap of 9 points is normal for GBM and expected. The model memorises training data but generalises well.
  • Best iteration at 312 — far fewer than the 1000 upper bound. Early stopping saved compute and prevented overfitting to the validation set.
  • Test AUC (0.871) ≈ Val AUC (0.874) — the model did not overfit to the validation set. Consistent performance on truly unseen data.
Stage 03

SHAP Values — The Explanation Layer

SHAP (SHapley Additive exPlanations) decomposes each individual prediction into the contribution of each feature. This is the tool that makes GBM governable.

  • Compute SHAP values on test set Run the SHAP explainer on the fitted GBM. Produces a matrix: one SHAP value per customer per feature. Positive SHAP = feature pushed prediction toward churn. Negative SHAP = pushed away from churn.SHAP values sum to the total prediction (log-odds): base value + sum(SHAP values) = final score. This property is called efficiency.
  • Summary plot: global feature importance Sort features by mean absolute SHAP value. This is the most reliable feature importance metric — unbiased, consistent with model behaviour. Colour shows whether high feature values push toward or away from churn.
  • Waterfall plot: individual explanation For a specific customer, show each feature's SHAP contribution. This becomes the script for a retention manager call or the adverse action reason code.

SHAP is the governance contractThe SHAP waterfall is what you show a regulator, a compliance auditor, or an affected customer when they ask 'why were you flagged?'. Ensure the top SHAP features are stable over time — a feature that drives predictions in Q1 but disappears by Q3 signals model drift.
Stage 04

QA — Good vs Red-Flag Output

Use this as a sign-off checklist before approving GBM for production use.

  • Check early stopping was used A GBM without early stopping is almost certainly overfit. Ask for the train vs validation AUC learning curve.
  • Verify SHAP features make business sense The top 5 SHAP features should be explainable in business terms to a non-technical stakeholder.
  • Confirm time-based validation split for any time-series-adjacent data If training data has a time dimension (transaction history, tenure), a random split leaks future information.
Output: QA Checklist Checklist
Good output
  • Early stopping used — best_iteration << n_estimators
  • SHAP features intuitive and consistent with domain knowledge
  • Val AUC ≈ Test AUC (within 3 points)
  • Calibration checked if probabilities used for decisions
  • Time-based split used for temporal data
Red flags
  • No early stopping — fixed n_estimators is a guess
  • AUC = 0.99+ on a real dataset → almost certain leakage
  • Top SHAP feature is customer_id, date, or an ID column
  • Training AUC >> Val AUC by more than 10 points → overfit
  • Random split used on data that has temporal ordering
