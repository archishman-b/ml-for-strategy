Stage 01

Why Random Forest over Logistic Regression

Before building a model, the analyst should articulate why they chose this approach. Random Forest is the right choice when: (1) you expect nonlinear relationships, (2) you have mixed feature types and don't want extensive preprocessing, or (3) you want a stable, hard-to-break baseline without tuning.

  • Compare against logistic regression baseline first Random Forest is more complex than logistic regression. Before using it, establish whether LR achieves acceptable performance. If LR AUC > 0.80 and interpretation matters, LR may be the better choice.Rule: always start with the simplest model that could plausibly work. Add complexity only when it demonstrably improves performance.
  • Identify the diversity mechanism Understand the two sources of diversity that make RF work: bootstrap sampling (each tree sees ~63% of data) and feature subsampling (max_features=√p features at each split). These ensure trees are different from each other, so averaging reduces variance.
  • Set up identical preprocessing pipeline Same feature set as Notebook 01, but scaling is optional for RF (trees are scale-invariant). Keep scaling for comparability. No imputation needed — sklearn RF handles missing values internally if desired.

Stage 02

Train & Tune Random Forest

Random Forest is remarkably robust to hyperparameter choices — the defaults work surprisingly well. The key parameters to tune are n_estimators (more is almost always better up to a point of diminishing returns) and optionally max_depth (leave unrestricted by default).

  • Train with n_estimators=300, max_features='sqrt' 300 trees is a safe starting point — performance typically plateaus after 200–500 trees. sqrt of p features at each split is the standard default for classification.Set random_state=42 for reproducibility. Different seeds will give slightly different results.
  • Use OOB score for free validation Out-of-bag score: each tree is trained on a bootstrap sample (~63% of data). The remaining 37% are scored without that tree. The OOB score aggregates these — a free, unbiased validation estimate without holding out a separate validation set.
  • Tune n_estimators using learning curve Train with 10, 50, 100, 200, 300, 500 trees. Plot OOB score vs n_estimators. Choose the 'elbow' — diminishing returns point. This is both a performance and compute cost decision.

Use permutation importance, not impurity importancesklearn's default feature importance (Mean Decrease Impurity / MDI) is biased toward high-cardinality features. A feature with 1000 unique values will look more important than it really is. Always use permutation importance (model.feature_importances_ via sklearn's permutation_importance function) for reliable driver analysis.
Stage 03

Calibration & Performance Summary

Random Forest probabilities are often reasonably well-calibrated but benefit from Platt scaling or isotonic regression before being used for expected value calculations.

  • Compare RF calibration against logistic regression LR is well-calibrated by default. RF tends to produce 'compressed' probabilities — fewer extreme scores (near 0 or 1). Check whether calibration matters for the intended use case.If you're only ranking customers (who to call first), calibration doesn't matter. If you need 'expected revenue at risk', calibration matters greatly.
  • Apply calibration if needed Wrap the RF in CalibratedClassifierCV(method='isotonic') and refit on training data. This corrects probability compression without affecting rank ordering.
