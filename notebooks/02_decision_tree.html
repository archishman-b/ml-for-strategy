Decision Tree

Business question: What decision rule separates this group? Output: Class label + readable rule path Dataset: customers.csv (800 rows) Use case: Adverse action reason codes · Ticket routing

This guide walks through the complete analysis pipeline for decision tree — from data to deployment-ready output. Each stage covers the workflow steps, followed by pre-populated example outputs and how to read them in a business context.

Stage 01

Data & Problem Setup

Decision trees are most valuable when you need an explainable rule path — for compliance, for policy prototyping, or for communicating segmentation logic. The dataset and target setup is identical to logistic regression.

  • Load data and define target Same setup as Notebook 01. Target: churned (1/0). Exclude customer_id. Numeric and categorical features.
  • Assess interpretability requirements Ask: what depth makes sense for the audience? A compliance officer needs depth ≤3 (8 terminal nodes max). A product team might accept depth 5.Rule of thumb: max 2^depth leaf nodes. Depth 3 = 8 leaves. Depth 5 = 32 leaves. Beyond that, a tree diagram is unreadable.
  • No preprocessing needed for trees Decision trees do not require feature scaling or imputation for splits. However, encoding categoricals is still required for sklearn's implementation.
Output: Feature Setup Console

Stage 02

Train Shallow Tree (Depth 3)

A depth-3 tree produces at most 8 leaf nodes and generates a fully readable decision flowchart. This is the governance version — suitable for adverse action notices and compliance documentation.

  • Fit DecisionTreeClassifier with max_depth=3 Train with max_depth=3, min_samples_leaf=10 (minimum 10 customers in any leaf), criterion='gini' (default). Fit on training data.min_samples_leaf=10 prevents tiny unstable leaves with unreliable probability estimates.
  • Visualise the tree Export and render the tree diagram. Each node shows: split rule, Gini impurity, sample count, class distribution.
  • Compare depth 2, 3, 4 on validation AUC Train three trees at different depths and compare AUC. The depth that adds <0.02 AUC over the previous depth is the right stopping point.

What this tells you
  • The most at-risk group: customers with tenure ≤ 24 months AND 3+ support calls. 68% churn rate among 89 customers — these are the top retention priority.
  • Long-tenure customers with high logins (top-right leaf) are the safest: 5% churn rate among 246 customers. These do not need retention investment.

  • This tree can be converted directly into a policy rule for a CRM system or call centre routing rule.
Stage 03

Reason Code Generation

The primary business value of a decision tree over more powerful models is its ability to generate an auditable reason path per customer — essential for compliance, adverse action notices, and collector scripts.

  • Extract decision path per customer For each customer, trace the path from root to leaf. Record each split decision as a plain-English rule.Output: 'Customer 1042: tenure=18 months (≤24) → support_calls=4 (≥3) → HIGH RISK (68%)'
  • Generate reason codes for flagged customers Convert each path to a ranked list of reasons. Reason 1 = the first split in the path (strongest discriminator). These become the CRM notes or adverse action text.

Stage 04

QA — Good vs Red-Flag Output

Sign-off checklist for decision tree outputs.

  • Check depth is appropriate for the audience A compliance auditor reading a tree must be able to trace any path manually. If the tree requires scrolling to see, it is too deep.
  • Verify leaf sample sizes Leaves with <10 samples produce unreliable probability estimates. Set min_samples_leaf accordingly.
  • Validate reason codes with domain experts Show the top 20 reason codes to a retention manager or underwriter. Any reason that 'doesn't make sense' is a red flag for data quality or feature construction issues.
Output: QA Summary Checklist
Good output
  • Tree depth ≤ 4 for compliance contexts
  • All leaves have ≥ 10 samples
  • Reason codes make intuitive business sense
  • Val AUC compared against random forest baseline
  • Depth chosen by AUC curve, not intuition
Red flags
  • Depth > 6 (unreadable, unstable)
  • Tiny leaf sizes (< 5 samples) — unreliable probabilities
  • AUC reported on training data only
  • Same split feature appears at every level (overfitting to one correlated feature)
  • Tree re-run and top splits change completely (instability)
