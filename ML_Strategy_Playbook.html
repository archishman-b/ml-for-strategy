Part 1 - Classification · 1.1

Logistic Regression

The gold standard for interpretable binary classification. Learns a linear decision boundary and converts it to a probability via the sigmoid function.

How it works

Predicts P(event) as a weighted sum of features, squashed through a sigmoid to produce a probability between 0 and 1.

P(y=1|x) = sigmoid(B0 + B1x1 + B2x2 + ... + Bnxn) sigmoid(z) = 1 / (1 + e^-z) Coefficient: exp(Bi) = odds multiplier for feature xi If exp(B) = 1.3: feature increases odds of event by 30%

Key properties

OutputProbability 0-1
DecisionThreshold (e.g. >0.5 = positive)
Training lossCross-entropy / log loss
RegularisationL1 (Lasso) or L2 (Ridge)
Feature scalingRequired - standardise inputs
Main pitfall to watch forClass imbalance makes accuracy misleading. Proxy features can encode protected attributes. Leakage from post-outcome variables is the most common failure mode.
When to escalateGradient Boosting if performance gap is significant after validation

Decision Tree

Splits data into progressively purer groups via if/then questions. Each split maximises information gain. The result is a readable flowchart.

How it works

At each node, choose the feature and threshold that most reduces uncertainty (entropy/Gini). Repeat until a stopping criterion (max depth, min leaf size) is met.

Split criterion: Information Gain = H(parent) - sum(w_i * H(child_i)) Entropy H(S) = -sum(p_i * log2(p_i)) [uncertainty measure] Leaf prediction: majority class (classification) or mean (regression) Key governance knobs: max_depth, min_samples_leaf

Key properties

OutputClass label + probability
DepthKey governance and interpretability knob
Min leaf sizeControls stability of leaf estimates
Feature scalingNot required
Handles mixed typesYes
Main pitfall to watch forThe most common mistake: growing trees too deep. A depth-10 tree has no business value as an explanation. Always visualise the tree before using it in communication.
When to escalateRandom Forest or GBM for performance; keep tree as surrogate explainer for communication

Part 1 - Classification · 1.3

Random Forest

Trains many decision trees on random data and feature subsets, then averages predictions. The diversity between trees reduces variance dramatically.

How it works

Bootstrap the data, randomly limit features at each split, build a tree. Repeat 100-500 times. Average the predictions. Uncorrelated errors cancel out.

Prediction = (1/T) * sum(f_t(x)) for T trees Key diversity mechanisms: 1. Bootstrap samples: each tree sees ~63% of data 2. Feature subsampling: max_features=sqrt(p) at each split OOB error: unbiased estimate using unused bootstrap samples

Key properties

OutputProbability / continuous
Key knobn_estimators, max_depth, max_features
OOB validationFree estimate without separate test set
Feature scalingNot required
ParallelismFully parallelisable training
Main pitfall to watch forFeature importance via mean decrease impurity is biased toward high-cardinality features. Use permutation importance or SHAP instead for reliable driver analysis.
When to escalateGradient Boosting for peak performance; stay with Random Forest when governance simplicity matters more

Part 1 - Classification · 1.4

Gradient Boosting (XGBoost / LightGBM)

Builds trees sequentially. Each new tree fits the residual errors of the current ensemble. Usually delivers best-in-class performance on tabular business data.

How it works

Start simple. Compute errors. Fit a tree to the errors. Add it with a small weight (learning rate). Repeat hundreds of times. Each iteration reduces the residual.

F_0(x) = initial prediction (mean) F_m(x) = F_{m-1}(x) + learning_rate * h_m(x) h_m = tree fitted to negative gradient of loss XGBoost adds L1/L2 regularisation on leaf weights LightGBM uses histogram-based leaf-wise splitting (faster)

Key properties

OutputProbability / continuous
Key knobsn_estimators, learning_rate, max_depth
Early stoppingPrevents overfitting automatically
Missing valuesHandled natively
Monotonic constraintsCan be enforced for business rules
Main pitfall to watch forThe most common failure: high AUC in the notebook, poor performance in production. Root causes: random splits on time-series data, feature leakage, distribution shift after deployment. Always validate with time-based splits.
When to escalateNeural Networks only for very large datasets or genuinely unstructured inputs

Naive Bayes

Applies Bayes theorem with the naive assumption that features are conditionally independent. Fast, simple, effective baseline - especially for text.

How it works

P(class|features) = P(class) x P(f1|class) x P(f2|class) x ... The independence assumption is usually wrong but the ranking is often surprisingly good.

P(y=k|x) proportional to P(y=k) * product(P(x_i | y=k)) Gaussian NB: P(x_i|y=k) = Normal(x_i; mean_k, var_k) Multinomial NB: for word counts and text data Bernoulli NB: for binary presence/absence features Training: just estimate means/variances per class - O(n) time

Key properties

OutputProbability
VariantsGaussian, Multinomial, Bernoulli
Training speedExtremely fast - O(n)
Feature scalingNot required
Text handlingMultinomial NB is a text classification standard
Main pitfall to watch forNaive Bayes is not a production workhorse for most business ML. Use as a baseline and for text problems. Probability outputs are often systematically miscalibrated when features are correlated.
When to escalateLogistic Regression for better-calibrated probabilities; GBM for better performance

Part 2 - Regression · 2.1

Ridge / Lasso Regression

Linear regression with regularisation. Ridge shrinks all coefficients; Lasso can zero them out (sparse model). The workhorse for driver analysis and CLTV baselines.

How it works

Adds a penalty to the loss function that discourages large coefficients. This prevents overfitting and makes coefficients more stable when features are correlated.

Ridge: minimise sum((y - Bx)^2) + lambda * sum(B_i^2) [L2 penalty] Lasso: minimise sum((y - Bx)^2) + lambda * sum(|B_i|) [L1 penalty] Standardised coefficients are directly comparable in magnitude Lasso zeros out weak features -> implicit feature selection Elastic Net: combination of L1 and L2

Key properties

OutputContinuous numeric value
CoefficientsDirectly interpretable (standardised)
Ridge vs LassoRidge: shrinks all. Lasso: zeros out weak features
Key hyperparameterAlpha (lambda) - regularisation strength
Feature scalingRequired for comparable coefficients
Main pitfallInterpreting raw (non-standardised) coefficients as importance is the most common mistake. A coefficient of 0.02 for revenue and 2.0 for channel index are not comparable without standardisation.
When to escalateGradient Boosting Regressor when nonlinear effects or interactions are important

Analytics

Operations

Strengths
  • Highly interpretable standardised coefficients
  • Handles multicollinearity well (Ridge)
  • Lasso performs implicit feature selection
  • Fast, stable, easy to govern
  • Strong baseline for all regression problems
Weaknesses

  • Sensitive to outliers (MSE loss)
  • Lasso unstable when features are correlated (picks one arbitrarily)
  • Requires explicit interaction terms

  • Was Ridge chosen over plain OLS? (Almost always yes for business data with correlated features)
  • Is the residual distribution examined? (Check for heteroscedasticity - errors that grow with the target value)
  • Is the target log-transformed if it is skewed? (Revenue, claims, spend are typically right-skewed)
  • Was Lasso used for feature selection? If so, are the selected features stable across CV folds?

Gradient Boosting Regressor

The regression version of gradient boosting. Sequentially fits trees to residual errors. Best-in-class when nonlinear effects and interaction terms matter.

How it works

Same engine as classification GBM but minimising a regression loss (MAE or MSE) rather than cross-entropy. Captures complex demand curves, nonlinear cost structures, and interaction effects.

Part 2 - Regression · 2.3

Quantile Regression

Predicts any percentile of the outcome distribution - not just the mean. Essential when the cost of over-prediction differs from under-prediction.

How it works

Standard regression minimises average squared error (MSE), implicitly predicting the mean. Quantile regression minimises an asymmetric loss that produces any target percentile.

Quantile loss for quantile q: L_q(y, y_hat) = q * |y - y_hat| if y > y_hat = (1-q) * |y - y_hat| if y <= y_hat Predicting Q90: the model is penalised 9x more for under-prediction Predicting Q10: the model is penalised 9x more for over-prediction Q50 = median regression (equivalent to MAE minimisation)

Key properties

OutputTarget percentile estimate
No distributional assumptionWorks with any outcome distribution
Key parameterQuantile q (e.g. 0.5, 0.75, 0.9)
Multiple modelsOne model per quantile
Linear versionQuantileRegressor (sklearn)
Main pitfallThe most common mistake: using mean predictions plus a fixed percentage buffer (e.g. mean + 20%) instead of estimating the actual target percentile. This is not equivalent and the buffer is usually wrong.
When to escalateGBR with quantile loss for nonlinear quantile estimation

Part 5 - Time Series · 5.1

Baseline Forecasting & ETS

Mandatory starting point. Naive and seasonal naive baselines set the performance floor. ETS decomposes trend and seasonality with exponential smoothing.

How it works

Before any model: seasonal naive (next period = same period last year) is often hard to beat. ETS adds trend and seasonal components with weights that decay for older observations.

Naive: y_hat(t+1) = y(t) Seasonal naive: y_hat(t+1) = y(t-m) where m = seasonality period ETS(A,A,A): Additive Error, Trend, Seasonality Level: l(t) = alpha * y(t) + (1-alpha) * (l(t-1) + b(t-1)) Trend: b(t) = beta * (l(t) - l(t-1)) + (1-beta) * b(t-1) Seasonal: s(t) = gamma * (y(t) - l(t)) + (1-gamma) * s(t-m)

Key properties

MandatoryAlways establish before any ML model
Seasonal naiveNext = same period last year - very hard to beat
ETSExponential smoothing with trend and seasonality
CI availableYes via bootstrapping or analytical
SpeedExtremely fast to train and serve
Main pitfallThe most dangerous mistake in forecasting: comparing an ML model against a naive/mean baseline rather than a seasonal naive baseline. ML improvements over seasonal naive are often much smaller - and sometimes negative.
When to escalateARIMA when confidence intervals and ACF/PACF diagnosis are needed; ML + lags when external regressors matter

Part 5 - Time Series · 5.2

ARIMA / SARIMA

Classical statistical time series model capturing autocorrelation, trend, and seasonality. Provides confidence intervals and is well-suited to univariate series.

How it works

ARIMA = AR (past values) + I (differencing to remove trend) + MA (past errors). SARIMA adds seasonal copies of each component. Together they model autocorrelated, trending, seasonal series.

ARIMA(p,d,q): y'(t) = phi_1*y'(t-1) + ... + phi_p*y'(t-p) + theta_1*e(t-1) + ... + theta_q*e(t-q) + e(t) where y'(t) = d-th difference of y(t) SARIMA(p,d,q)(P,D,Q,m): adds seasonal AR, differencing, MA m = seasonality period (12 for monthly, 7 for daily) Diagnostics: ACF/PACF plots -> select p, q ADF test -> check stationarity Ljung-Box test -> check if residuals are white noise

Key properties

OutputForecast + confidence intervals
Key parametersp,d,q (and P,D,Q,m for seasonal)
StationarityRequired - difference to achieve
CIYes - analytical confidence intervals
SelectionAIC/BIC for parameter selection
Main pitfallThe most common mistake with ARIMA: applying it to a non-stationary series without differencing first. Always run the ADF test. Also: checking residual diagnostics is not optional - if residuals are autocorrelated, the model has not captured all the signal.
When to escalateML + lag features when external regressors (promotions, price, weather) drive the series

Part 6 - Specialised · 6.1

Survival Analysis

Models the time until an event occurs, correctly handling censoring: customers who have not yet churned are still informative data points.

How it works

In standard regression, you ignore customers still active (censored). Survival analysis uses them correctly: a customer with 24 months tenure who has not churned tells you they survived at least 24 months.

Survival function: S(t) = P(T > t) - probability of surviving past time t Kaplan-Meier: non-parametric step function estimate of S(t) S(t_j) = product_{t_i <= t_j} (1 - d_i/n_i) d_i = events at time t_i, n_i = at-risk at time t_i Cox PH model: h(t|x) = h_0(t) * exp(B^T * x) h_0(t) = baseline hazard exp(B_i) = hazard ratio for feature x_i

Key properties

Key inputDuration column + event indicator (1=churned, 0=censored)
Kaplan-MeierNon-parametric survival curve - no feature covariates
Cox PHSemi-parametric - adds feature effects via hazard ratios
Hazard ratioexp(B) = how much faster/slower the event happens
CensoringStill-active customers are included - not ignored
Main pitfallThe most common error: using standard logistic regression for time-to-event data. This ignores censoring - you effectively exclude all currently active customers, introducing severe selection bias. Use Kaplan-Meier or Cox PH instead.
When to escalateRandom survival forests or DeepSurv for complex nonlinear survival modelling

Part 6 - Specialised · 6.2

Uplift Modelling (T-Learner)

Estimates who will respond BECAUSE OF your intervention - not just who will respond. Maximises campaign ROI by targeting only Persuadables.

How it works

A response model asks 'who will convert?'. An uplift model asks 'who will convert because of our action?' The difference is critical: targeting Sure Things wastes budget; targeting Do-Not-Disturb customers causes harm.

Individual uplift: tau(x) = P(Y=1 | T=1, X=x) - P(Y=1 | T=0, X=x) T-Learner (two models): 1. Train model_1 on treated group: estimate P(Y=1|T=1, x) 2. Train model_0 on control group: estimate P(Y=1|T=0, x) 3. Uplift = model_1.predict(x) - model_0.predict(x) Requires: randomised treatment assignment (A/B test data)

Key properties

RequiresRandomised experiment (A/B test) data
OutputIndividual uplift score per customer
Four segmentsPersuadable, Sure Thing, Lost Cause, Do Not Disturb
ValidationQini curve, uplift curve
T-LearnerTwo separate response models - simplest approach
Main pitfallThe fundamental problem of causal inference: you can never observe the counterfactual for an individual. A customer either received the treatment or did not - never both. Uplift estimates are always noisy. Validate on held-out experiment data with Qini curves, not just AUC.
When to escalateX-Learner or doubly robust estimators for better performance; meta-learners for complex treatment effect heterogeneity

Real consulting use case scenarios

CRM
Retention campaign targeting
Budget to contact 10% of churners. Uplift model identifies the 10% most likely to change behaviour due to contact - not just the 10% most likely to churn.
Marketing
Discount offer optimisation
Who needs a discount to convert, vs who will convert anyway? Uplift model targets only the former, dramatically improving offer ROI.
Collections
Proactive contact prioritisation
Which overdue accounts will pay earlier due to proactive contact? Uplift targets these rather than accounts that will pay regardless.
Strengths
  • Maximises campaign ROI directly
  • Avoids wasting budget on Sure Things
  • Identifies and protects Do-Not-Disturb customers
  • Individual-level treatment effect estimates
  • Directly actionable - rank and target Persuadables
Weaknesses
  • Requires randomised experiment data
  • Uplift cannot be directly observed at individual level (fundamental problem)
  • Noisier than standard response models
  • Larger samples needed for reliable estimates
  • Complex to explain to stakeholders unfamiliar with causal framing

Questions to ask your data science team

  • Was the training data from a randomised experiment with proper control and treatment groups?
  • Is the uplift curve (Qini curve) showing improvement over random targeting?
  • What is the estimated budget saving from targeting Persuadables only vs blanket campaign?
  • How are Do-Not-Disturb customers identified and protected from contact?

Anomaly Detection

Identifies observations that are unusual given the expected pattern. Labels are rare or absent - the challenge is defining 'normal' without knowing all the ways abnormal can appear.

How it works

Three tools: Z-scores flag extreme individual feature values; Isolation Forest catches multivariate anomalies; Autoencoders learn complex normal patterns and flag what does not reconstruct well.

Z-score: z = |x - mu| / sigma -> flag if z > threshold (2, 3, 4) Isolation Forest: anomaly score = 2^(-E(h(x))/c(n)) h(x) = average path length to isolate x Anomalies require fewer splits to isolate Autoencoder: reconstruction error = ||x - decoder(encoder(x))||^2 Train on NORMAL data only High error = anomalous (does not fit learned normal pattern)

Key properties

Z-scoreUnivariate, fast, auditable, assumes normality
Isolation ForestMultivariate, no labels, scales well
AutoencoderComplex patterns, trains on normal data, needs more tuning
ThresholdSet based on investigation capacity - not just accuracy
Ground truthOften not available - validate against known cases if any
Main pitfallAlert fatigue is the silent killer of anomaly detection programmes. A model that flags 5% of transactions for a team that can investigate 0.5% is 90% wasteful and will be ignored within weeks. Always size the threshold to investigation capacity first.
When to escalateLabelled anomalies -> train a classifier (GBM) on confirmed cases for much better performance

Real consulting use case scenarios

Finance
Procurement fraud detection
Flag invoices with unusual amount, frequency, vendor, or timing combinations. Isolation Forest catches the combinations that z-score misses.
Insurance
Claims leakage identification
Flag claims with unusual combinations of injury type, treatment duration, and medical provider. Each flag goes to a special investigations unit.
Ops
Production line anomalies
Monitor 100 sensor readings. Isolation Forest flags unusual combinations of readings that individually look normal but collectively signal a developing fault.
Risk
AML transaction monitoring
Flag transactions with unusual patterns: unusual size for account type, unusual counterparty, unusual timing. Feed to human review queue.
Strengths
  • No anomaly labels required (unsupervised)
  • Z-scores are fully auditable and explainable
  • Isolation Forest catches multivariate anomalies
  • Can find novel anomaly types not seen before
  • Scalable to very large volumes
Weaknesses
  • Threshold tuning is subjective
  • High false positive rates if not calibrated to capacity
  • Hard to explain individual flags in detail
  • Cannot distinguish anomaly types automatically
  • Ground truth for validation is often scarce

Questions to ask your data science team

  • What is the target investigation rate? (Set threshold based on capacity - not on maximising recall in isolation)
  • How are flagged cases explained to investigators? (Z-scores are auditable; Isolation Forest scores are not directly interpretable)
  • Was the model validated against any known anomaly cases?
  • How does the alert volume compare to investigation capacity? (An overcapacity alert system is worthless)

A/B Testing & Causal Inference

The gold standard for measuring causal impact of an intervention. Randomisation ensures the two groups are comparable on everything except the treatment.

How it works

Randomly assign eligible units to treatment (new policy) or control (current policy). Measure primary KPI and guardrails. The difference is the Average Treatment Effect (ATE) - causal, not correlational.

ATE = E[Y | T=1] - E[Y | T=0] Statistical test: t-test (continuous), z-test (proportions) p-value: P(observing result this extreme | no true effect) 95% CI: [ATE - 1.96*SE, ATE + 1.96*SE] Minimum Detectable Effect (MDE): n_per_group = 2 * ((z_alpha + z_beta) * sigma / delta)^2 alpha=0.05, power=0.8 -> z values = 1.96, 0.84

Key properties

Gold standardRandomisation ensures causal interpretation
Primary outputATE + 95% confidence interval
Pre-registerDefine metrics before looking at data
DurationRun for at least 1 full business cycle
Sample sizePower analysis determines minimum sample
Main pitfallPeeking: stopping the test as soon as p < 0.05 inflates false positives dramatically. Pre-commit to a sample size via power analysis and read results only once at the predetermined end. If you must peek, use sequential testing methods.
When to escalateUplift modelling when you want individual-level treatment effect estimates after the A/B test

Real consulting use case scenarios

Product
Model threshold test
Test a new churn model threshold: treatment gets new threshold, control gets current. Primary: intervention rate. Guardrail: churn rate at 90 days.
Pricing
Discount offer test
Treatment receives targeted discount, control receives standard offer. Primary: revenue per customer. Guardrail: margin and complaint rate.
CX
Contact strategy test
Test proactive outreach for at-risk customers. Treatment: contact at day 30. Control: no contact. Primary: 90-day retention. Guardrail: NPS.
Strengths
  • Cleanest causal estimate available
  • Well-understood statistical framework
  • Easy to explain to executives
  • Can test policies, thresholds, and model deployments
  • Guardrails detect unintended harms
Weaknesses
  • Requires randomisation (not always feasible)
  • Takes time to accumulate sufficient sample size
  • Interference effects can bias results
  • Early stopping inflates false positives
  • Cannot estimate individual effects (only average)

Questions to ask your data science team

  • Was the unit of randomisation correct? (User, account, store, or market - depends on where interference can occur)
  • Was a power analysis done before the test to determine required sample size?
  • Were success metrics and guardrails defined before looking at any results?
  • Was the test run for at least one full business cycle (e.g. 2 full weeks for weekly-seasonal data)?
  • Was a Sample Ratio Mismatch (SRM) check done in the first 24 hours?

Neural Networks & Deep Learning

Learns hierarchical feature representations through layers of nonlinear transformations. The default choice for unstructured data; often overkill for structured tabular business data.

How it works

Each layer transforms inputs into increasingly abstract representations. The network learns which transformations minimise prediction error. Backpropagation computes gradients; gradient descent updates weights.

Layer l: h_l = activation(W_l * h_{l-1} + b_l) Common activations: ReLU, Sigmoid, Tanh, Softmax Loss: Cross-entropy (classification), MSE (regression) Optimisation: SGD, Adam, RMSprop Regularisation: Dropout, L2 weight decay, Early stopping, Batch norm Deep learning beats classical ML when: - Data > 100K rows (ideally millions) - Input is unstructured (text, image, audio, sequence) - Features are difficult to engineer manually

Key properties

Default forUnstructured data: text, images, audio, sequences
vs GBM on tabularGBM usually wins unless data is very large
ComputeOften requires GPU for non-trivial architectures
InterpretabilityLow - SHAP or LIME for post-hoc
Governance effortHigh - more complex validation and monitoring required
Main pitfallUsing deep learning for structured tabular data with fewer than 100K rows. In practice, well-tuned GBM outperforms vanilla neural nets on most business tabular datasets. Deep learning adds governance overhead without performance benefit in these cases.
When to escalateTransformers for text/sequence tasks; CNNs for image tasks

Real consulting use case scenarios

BERT-based classifier extracts themes and sentiment from 100K+ NPS comments. Not achievable with classical NLP at this quality.

CNN-based OCR extracts structured fields (date, amount, vendor) from scanned invoices. Replaces manual data entry.
Sequence
Transaction sequence fraud detection
LSTM or Transformer processes the sequence of a customer's last 30 transactions. Detects complex sequential fraud patterns.
Strengths
  • Best performance on unstructured data (text, images)
  • Learns features automatically from raw inputs
  • Can handle very high-dimensional inputs
  • Transfer learning reduces data requirements
  • Scales to arbitrarily large datasets
Weaknesses
  • Often loses to GBM on structured tabular data
  • Requires large amounts of data
  • Computationally expensive
  • Low interpretability by default
  • Complex hyperparameter tuning and architecture design

Questions to ask your data science team

  • What architecture was chosen and why? Has it been validated against a GBM baseline?
  • How many parameters does the model have, and is it appropriate for the dataset size?
  • What is training time, inference latency, and serving infrastructure cost?
  • How are explanations generated, and have they been validated for stability?

Transformers & Foundation Models (LLMs)

Self-attention mechanisms that learn which parts of a sequence are relevant to each other. Power GPT, Claude, BERT, and virtually all modern NLP and AI systems.

How it works

Transformers process entire sequences simultaneously via self-attention - learning relationships between any pair of tokens regardless of distance. This enables understanding of complex context and meaning.

Self-attention: Attention(Q,K,V) = softmax(QK^T / sqrt(d_k)) * V Query Q, Key K, Value V = linear projections of input Attention weights show which tokens attend to which Deployment patterns (choose based on volume/latency/accuracy): Zero-shot API: OpenAI/Anthropic API, general tasks Few-shot: Prompt with 3-10 examples in context RAG: Retrieve + generate, grounded in your docs Fine-tuning: Train on domain data, best accuracy Embeddings: Dense vectors for similarity/search

Key properties

Zero-shotAPI call, no training, general tasks - start here
RAGGround LLM in your own documents - reduces hallucination
Fine-tuningBest accuracy, higher cost, needs labelled data
EmbeddingsSemantic similarity, clustering, search
GovernanceHallucination, prompt injection, PII leakage - all require controls
Main pitfallThe most dangerous LLM pitfall: no hallucination mitigation strategy. LLMs will confidently state incorrect facts, produce plausible-sounding but wrong analysis, and fail silently. Always implement: grounding (RAG), fact-checking loops, uncertainty disclosure, and human review for high-stakes outputs.
When to escalateFine-tuning for high-volume/latency-sensitive tasks; RAG for document-grounded tasks; embeddings for similarity/search

Real consulting use case scenarios

Analytics
Earnings call / analyst report summarisation
Summarise 20-page analyst reports into 3-paragraph executive briefs. RAG grounds the summary in the actual document.

Complaint theme extraction
Classify 50K support tickets by theme, sentiment, and urgency. Zero-shot GPT-4 often matches fine-tuned BERT at a fraction of the engineering effort.
Legal
Contract clause extraction
Extract key terms (termination clauses, liability caps, renewal dates) from thousands of contracts. RAG with a fine-tuned extractor.
Knowledge
Internal knowledge base search
Embed all internal documents. Semantic search lets users query in natural language and get the most relevant sections. RAG powers the answers.
Strengths
  • Exceptional text understanding and generation
  • Zero-shot capability on many tasks (no training data)
  • Rapidly improves with prompt engineering
  • Multi-modal capabilities (text + image)
  • Embeddings power semantic search and clustering
Weaknesses
  • Hallucination - confident incorrect outputs
  • Non-deterministic - same input may give different outputs
  • Cost at scale - token pricing can be significant
  • Latency - API calls add 200ms-2s per request
  • Governance complexity - PII, prompt injection, output quality

Questions to ask your data science team

  • Why a foundation model vs a fine-tuned classifier for this task? Has a simpler approach been tried?
  • How is hallucination detected and handled? What happens when the model is confidently wrong?
  • What is the cost and latency per call at expected production volume?
  • How is PII handled in the prompt? Who has access to the inference logs?
  • What is the evaluation framework? (Human eval + automated tests - 'it sounds good' is not a metric)

§8Evaluation & Metrics

Choosing the wrong metric is one of the most common ways a model looks great in the notebook and fails in production. This section gives you the vocabulary to have that conversation with your data science team.

Classification Metrics

The confusion matrix is the foundation of all classification metrics. Always ask to see it — not just a single number.

The Confusion Matrix — what every metric is built from
Accuracy
(TP+TN)/(TP+TN+FP+FN). Proportion correct overall.
⚠ Useless with class imbalance. 99% accuracy on 1% fraud rate = always-predict-no model.
Precision
TP/(TP+FP). Of all flags raised, what fraction were real?
Use when false alarms are costly: investigation queues, clinical referrals.
Recall (Sensitivity)
TP/(TP+FN). Of all real positives, what fraction did we catch?
Use when misses are costly: fraud, disease, safety issues.
F1 Score
2 × (Precision × Recall)/(Precision + Recall). Harmonic mean.
Balanced view when both FP and FN matter. Use F-beta to weight one more.
AUC-ROC
Probability that model ranks a random positive above a random negative. Threshold-free.
Use to compare model versions. Not useful for communicating business impact.
Precision@K
Precision of the top K predicted positives. Relevant when you have a fixed action capacity.
Use when you can only action N cases per day. What fraction of those N are real?
The Threshold Question AUC measures ranking quality. The threshold determines the operating point (precision vs recall tradeoff). Always choose the threshold based on the business cost of FP vs FN — not by defaulting to 0.5. Ask: "What does it cost us when we flag a non-fraudster? What does it cost us when we miss a fraudster?"

Regression Metrics

MAE
Mean Absolute Error. Average size of error in original units.
Robust to outliers. Directly interpretable. Use when outlier errors are not more important than typical errors.
RMSE
Root Mean Square Error. Penalises large errors more than MAE.
Use when large errors are disproportionately costly. Sensitive to outliers — check distribution of errors.
MAPE
Mean Absolute Percentage Error. Scale-free, easy to communicate.
Use for forecasting comparisons across different scales. Breaks when actuals are near zero.
R²
Proportion of variance explained. 1.0 = perfect, 0 = baseline mean.
Good for driver analysis. Not interpretable in isolation — always compare to baseline R².

Forecasting Metrics

The Baseline Rule Every forecasting metric must be compared against a seasonal naive baseline. If your model achieves MAPE of 12% but seasonal naive achieves 10%, the model is adding complexity without value. The question is never "is 12% MAPE good?" — it's "is 12% better than naive by enough to justify the model?"

Clustering Validation

Silhouette Score
How similar each point is to its own cluster vs nearest neighbour cluster. Range: -1 to 1.
>0.5 = good separation. Use to select K alongside elbow method.
Business Validation
Can you name each cluster in business terms? Can you define a distinct action for each? Is the split stable over time?
The most important cluster validation. Statistical metrics are necessary but not sufficient.
BIC (GMM)
Bayesian Information Criterion. Penalises model complexity. Lower = better. Use to select K for GMM.
Plot BIC vs K and choose the K at the elbow or minimum.

§9Pre-Launch Checklist

For every model before it goes live. If you are reviewing a data science team's work, this is your sign-off framework. These are the questions that separate models that work in production from models that work in notebooks.

Phase 1
Data & Problem
  • 01 Is the target variable clearly defined and does it match the business question?
  • 02 Is every feature available at the time the model will make its decision? (No leakage)
  • 03 Is the training data representative of the population the model will score?
  • 04 Was the validation split done correctly? (Time-based for time series; no look-ahead bias)
  • 05 Is class imbalance addressed? (Metrics, thresholds, and sampling strategy documented)
Phase 2
Model Performance
  • 06 Is performance compared against a simple baseline? What is the lift, and is it business-meaningful?
  • 07 Is the right metric used? (Recall for fraud, Precision@K for capacity-constrained ops, MAPE for forecasts)
  • 08 For classification: are probabilities calibrated? (Reliability diagram shows actual vs predicted rates)
  • 09 Is the threshold set based on business cost of FP vs FN — not defaulted to 0.5?
  • 10 Is performance consistent across key subgroups? (Products, channels, geographies, demographics)
Phase 3
Explainability
  • 11 Do the top features make business sense? (A feature you cannot explain should not be in the model)
  • 12 Are the top features stable across retraining runs and time windows?
  • 13 Are any features proxies for protected attributes? (Postcode as proxy for ethnicity, etc.)
  • 14 Can you explain a specific prediction to an affected customer or an auditor?
  • 15 Are feature direction effects (positive/negative) aligned with domain knowledge?
Phase 4
Deployment
  • 16 Is the model served via a reproducible pipeline — not a manual notebook run?
  • 17 Is the model versioned? Can you roll back to the previous version if needed?
  • 18 What is the latency at P99 under expected production load? Is it within SLA?
  • 19 What does the model do when input data is missing or out-of-range? Is this handled gracefully?
  • 20 Is the action downstream of the model tested? (Model output → business action loop validated)
Phase 5
Monitoring
  • 21 Is input feature distribution monitored? (PSI or KS test — data drift detection)
  • 22 Is prediction distribution monitored? (Score distribution shift is often the first signal of model degradation)
  • 23 When outcomes are available, is actual vs predicted performance tracked on a dashboard?
  • 24 Is there a retraining schedule — and is it triggered by performance degradation, not just calendar time?
  • 25 Is there an owner responsible for model health — not just the initial data scientist who built it?
Phase 6
Governance
  • 26 Is the model documented? (Problem statement, data lineage, features, validation results, limitations)
  • 27 Are decisions made by the model subject to any regulatory or fairness requirements?
  • 28 Is there a human override mechanism? Can a customer or employee be escalated out of the model's decision?
  • 29 Was a shadow deployment run before full rollout? (Model ran silently, output not acted on, performance checked)
  • 30 Is business impact measurement in place? (How will you know in 6 months if the model delivered ROI?)
The 30-question test A model that cannot pass this checklist is not production-ready — regardless of how high its AUC is. The checklist is also a useful tool for scoping the data science engagement: which of these 30 questions are in scope, and who owns each one?
